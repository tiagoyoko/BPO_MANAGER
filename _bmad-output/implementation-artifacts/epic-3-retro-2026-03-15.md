## 2. Problemas e Lições Aprendidas

### P1 — Débito TypeScript acumulado silenciosamente
**O que aconteceu:** `tsc --noEmit` revelou 10 erros de tipo ao final do epic. Testes passavam (Vitest não checa tipos), mas o compilador não confiava nos tipos. Os erros incluíam casts incorretos de join do Supabase (`as Type` em vez de `as unknown as Type`) e atribuição a `process.env.NODE_ENV` read-only.

**Lição:** Rodar `tsc --noEmit` deve ser parte do Definition of Done de cada story — não apenas no final do epic.

**Ação para Epic 4:** Adicionar `tsc --noEmit` ao checklist de code review de cada story.

---

### P2 — Decisões de escopo tomadas durante implementação
**O que aconteceu:** Em 3-3 (timeline), a decisão de usar `isInternalBPOUser` (RBAC semântico) vs `canAccessModelos` foi tomada durante o code review, não no planejamento. O padrão de dupla query também não foi antecipado no design.

**Lição:** Endpoints que cruzam múltiplas tabelas (timeline, indicadores) merecem um mini-design técnico antes da implementação.

**Ação para Epic 4:** Para stories com múltiplos joins ou lógica de acesso complexa, criar uma nota de design antes de abrir o worktree.

---

### P3 — Tipo Supabase join: array vs objeto único
**O que aconteceu:** Supabase retorna joins como `{ field: any }[]` (array), mas tipos locais definem `{ field: string } | null`. Isso causou TS2352 em múltiplos arquivos — o padrão correto (`as unknown as Type`) não estava documentado.

**Lição:** Padrão de cast para joins Supabase deve estar em um lugar de referência rápida (CLAUDE.md ou convenções técnicas).

**Ação para Epic 4:** Documentar o padrão `as unknown as Type` para joins Supabase no CLAUDE.md do projeto.

---

### P4 — Segurança de tokens e segredos (antecipando Epic 8)
**Risco identificado:** A story 8-4 (cofre de senhas) lida com criptografia de tokens. Se o padrão de armazenamento não for definido cedo, pode haver débito de refatoração retroativo nos tokens já armazenados (ex.: token F360 de Epic 4).

**Ação para Epic 4:** Definir e documentar o padrão de armazenamento de segredos antes de implementar a integração F360, para não precisar migrar depois.
